# Arquitectura y Buenas Prácticas en React Moderno

Cuando construimos aplicaciones en React que superan las 10,000 líneas de código, la falta de una arquitectura sólida se convierte en el mayor enemigo del equipo. Atrás quedaron los días de componentes de clase gigantes; hoy dominan los **Hooks**, la inmutabilidad y los componentes funcionales puros.

En este artículo, exploraremos las prácticas que separan a un desarrollador Junior de un Senior en el ecosistema de React.

<Callout type="info" title="El Principio de Responsabilidad Única">
Mantén tus componentes pequeños, puros y enfocados en hacer una sola cosa bien. Si un componente hace fetch de datos, renderiza una tabla, y además maneja la validación de un formulario, está haciendo demasiado.
</Callout>

## 1. Separación de Lógica y Presentación (Custom Hooks)

El error más común en aplicaciones React es el "Componente Gordo" (Fat Component). Estos son componentes que mezclan la lógica de negocio (peticiones HTTP, formateo de datos, cálculos) con la interfaz de usuario (HTML/JSX).

La solución es extraer toda esa lógica a **Hooks Personalizados**.

### ❌ El Enfoque Incorrecto (Fat Component)

```tsx
export function UserDashboard() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    async function fetchUsers() {
      try {
        const response = await fetch('/api/users');
        const data = await response.json();
        setUsers(data);
      } catch (err) {
        setError(err.message);
      } finally {
        setLoading(false);
      }
    }
    fetchUsers();
  }, []);

  if (loading) return <Spinner />;
  if (error) return <ErrorAlert message={error} />;

  return (
    <div className="grid">
      {users.map(user => <UserCard key={user.id} data={user} />)}
    </div>
  );
}
```

### ✅ El Enfoque Correcto (Lógica Abstraída)

Primero, creamos nuestro hook personalizado:

```tsx
// hooks/useUsers.ts
export function useUsers() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    async function fetchUsers() {
      try {
        const response = await fetch('/api/users');
        const data = await response.json();
        setUsers(data);
      } catch (err) {
        setError(err.message);
      } finally {
        setLoading(false);
      }
    }
    fetchUsers();
  }, []);

  return { users, loading, error };
}
```

Y nuestro componente queda limpio y enfocado exclusivamente en renderizar UI:

```tsx
// components/UserDashboard.tsx
import { useUsers } from '../hooks/useUsers';

export function UserDashboard() {
  const { users, loading, error } = useUsers();

  if (loading) return <Spinner />;
  if (error) return <ErrorAlert message={error} />;

  return (
    <div className="grid">
      {users.map(user => <UserCard key={user.id} data={user} />)}
    </div>
  );
}
```

## 2. El Manejo del Estado en 2024

El ecosistema de estado global ha evolucionado drásticamente. Redux solía ser el rey indiscutible, pero su configuración excesiva lo ha relegado a casos de uso muy específicos.

Nuestra regla de oro para el estado es la siguiente:

1. **Estado del Servidor (Caché):** Utiliza librerías como `React Query` o `SWR`. Estas herramientas manejan peticiones, caché, reintentos y paginación automáticamente. No guardes datos que vienen de una API en un estado global manual.
2. **Estado de la UI Global (Tema, Sidebar, Autenticación):** Utiliza `Zustand`. Es infinitamente más ligero que Redux y no sufre de re-renders masivos como el `React Context` nativo.
3. **Estado de Formularios:** Utiliza `React Hook Form` combinado con `Zod` para validaciones.

<Callout type="warning" title="El problema del Context API">
El `Context API` de React es excelente para inyección de dependencias (como el tema oscuro/claro), pero es terrible para estado que cambia frecuentemente. Si un valor en el Context cambia, TODOS los componentes que lo consumen se volverán a renderizar, independientemente de si usan ese valor específico o no.
</Callout>

## 3. Optimización de Rendimiento (Memoization)

React es excepcionalmente rápido por defecto. Sin embargo, en listas largas o componentes pesados (como gráficos), los re-renders innecesarios pueden congelar el navegador.

Aquí es donde entran `useMemo` y `useCallback`.

### Cuándo usar useMemo

Utilízalo para evitar repetir cálculos costosos matemáticos o filtrados de arrays masivos en cada render.

```tsx
function DataGrid({ data, filterTerm }) {
  // ✅ Correcto: Solo recalcula si la data o el filtro cambian
  const filteredData = useMemo(() => {
    console.log("Calculando filtrado pesado...");
    return data.filter(item => item.name.includes(filterTerm));
  }, [data, filterTerm]);

  return <Table data={filteredData} />;
}
```

### Cuándo NO usarlos

No envuelvas cada variable en `useMemo`. El proceso de recordar (memoizar) un valor también cuesta memoria y CPU. Si la operación es simple, `useMemo` hará que tu aplicación sea *más lenta*.

## Conclusión

El código limpio en React no se trata de usar los trucos más oscuros o las funciones más nuevas. Se trata de legibilidad, predictibilidad y separación de preocupaciones. Si mantienes tus componentes enfocados y delegas la lógica compleja a Hooks, tu base de código podrá escalar sin límites.
